# 文档维护约定

这套文档会被多个 AI agent app 共同读取和逐步维护，例如 Codex、Cursor、Claude Code、Antigravity 等。目标是让稳定的项目知识沉淀在工作区内，而不是散落在某个 app 私有记忆、规则页或单次对话中。

## 总原则

- 发现稳定事实就回写文档，不要只停留在当前回答。
- 优先复用现有文档；只有现有文件承载不了时，才新增专题文件。
- 写入内容要面向所有 agent app 可读，不要依赖某个产品专属格式、按钮、工作流或私有状态。
- 每次发生稳定文档更新后，都要顺手追加一条变更日志，不要只改正文不留痕。

## 写到哪里

- 全局默认规则、项目结构现状、跨专题共性约束：写到 `ProjectAgents/ProjectAgents.md`
- 某一类专题知识，例如模块地图、检索手法、风险清单：写到 `ProjectAgents/references/` 下对应文件
- 新出现且会反复使用的专题：新增 `references/*.md`，并在 `ProjectAgents.md` 里补入口

## 适合回写的内容

- 项目结构发生稳定变化，例如新增 / 拆分 module、sourceSet 变化、默认按外部依赖处理的模块形态变化
- 高频检索误区，例如某类大文件禁止全量读取
- 稳定的风险检查项、验证策略、跨模块约束
- 多次重复出现、且能抽象成通用规则的实现偏好或阅读顺序

## 不适合回写的内容

- 只对当前一次任务有效的临时结论
- 未验证的猜测
- 强依赖某个 app 私有能力的操作提示
- 很快会过时、且没有稳定模式的偶发现象

## 更新方式

- 小改动直接补充现有段落，保持文档连续性
- 大改动优先先更新专题文件，再在 `ProjectAgents.md` 里加一句入口说明
- 如果多个文件都要改，确保 `ProjectAgents.md` 仍然是总入口，不要让信息只存在于深层子文件
- 如果目标项目把 `ProjectAgents/` 或模板目录加入了 `.gitignore`，确认这些文档是否已修改时，不要只看根目录 `git status`，必要时直接查看文件内容或用 `git check-ignore` 解释原因
- 更新完成后，追加记录到 `ProjectAgents/CHANGELOG.md`

## 变更日志约定

统一使用 `ProjectAgents/CHANGELOG.md` 记录文档改动，目标是让多人协同时至少能快速回答：
- 什么时候改的
- 哪个 app 改的
- 这次是谁提交 / 执行的
- 改了哪些文件
- 大致改了什么

记录规则：
- 只追加，不回改历史记录，除非修正明显事实错误
- 每次改动只写一条，控制在简短几行内，重点说变化，不要复制大段正文
- 优先记录稳定文档改动，不要把纯临时试验也写进去
- 如果多人在不同分支同时追加，合并时保留双方记录，不要二选一覆盖

每条记录至少包含：
- 时间：本地时间，写全日期、时间
- App：例如 `Codex`、`Cursor`、`Claude Code`、`Antigravity`
- Git 提交人：优先 `git config user.name`；需要时可附 `user.email`
- 涉及文件：本次改到的文档路径
- 摘要：1 到 5 条，概括本次关键变化

建议使用固定模板，减少多人协同时的格式漂移：

```md
## YYYY-MM-DD HH:MM:SS | AppName | git: user

- Files: `path/a.md`, `path/b.md`
- Summary: 调整了什么
- Summary: 补充了什么
```

## 写作要求

- 用短句写清规则和触发条件
- 明确“什么时候适用”和“什么时候不要这么做”
- 尽量给出具体文件名、模块名、命令模式或反例
